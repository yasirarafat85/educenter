## প্রজেক্ট কী

**EduCenter** — একটা বাংলা শিক্ষা প্রতিষ্ঠানের ওয়েবসাইট (কোর্স, ওয়ার্কশিট, প্রোডাক্ট বিক্রি করে), সম্পূর্ণ কাস্টম-বিল্ট Native PHP + MySQL এ, কোনো framework/Composer/SSH ছাড়া (যাতে সাধারণ শেয়ার হোস্টিং এ সরাসরি চলে)। ইউজারের কোনো কোডিং জ্ঞান নেই — সব কনটেন্ট ম্যানেজমেন্ট Admin Panel থেকেই হয়।

> **🔴🔴 ZIP তৈরির নিয়ম (২০২৬-০৭-১৬ এ শেখা, গুরুত্বপূর্ণ)**: cPanel/Linux-এ আপলোডের ZIP **কখনো** Windows PowerShell-এর `Compress-Archive` বা `.NET Framework ZipFile::CreateFromDirectory` দিয়ে বানাবেন না — এগুলো এন্ট্রি-পাথে **backslash** (`admin\includes\.htaccess`) লেখে, যা Linux/cPanel এক্সট্রাক্টে nested ফোল্ডার না বানিয়ে **ভুল নামে রুটে ফাইল ফেলে** (nested ফাইল নীরবে ডিপ্লয় হয় না — একটা পুরো লাইভ ডিপ্লয় আংশিক ভেঙেছিল এভাবে)। এছাড়া `Compress-Archive -Path $files.FullName` স্ট্রাকচার **flatten** করে দেয়। **সঠিক উপায়**: Python `zipfile` দিয়ে `arcname` এ `os.sep→'/'` রিপ্লেস করে (`z.write(full, rel.replace(os.sep,'/'))`) — forward-slash নিশ্চিত। যাচাই: `any(chr(92) in nm for nm in namelist())` → False হতে হবে।
> **🔴 আরও একটা ZIP ফাঁদ (২০২৬-০৭-২০ এ লাইভে ধরা পড়েছে)**: `z.writestr('path', 'content')` এ শুধু স্ট্রিং দিলে entry-র `external_attr = 0` থাকে — Linux/cPanel এক্সট্রাক্টে ফাইল/ফোল্ডার **পারমিশন 0000** নিয়ে তৈরি হয়, ফলে PHP আর ঐ ফোল্ডারে কিছু লিখতে পারে না (uploads/worksheets এভাবে তৈরি হয়ে ছবি আপলোড ভেঙেছিল; মজার ব্যাপার — `is_writable()` তবু true বলছিল)। **সঠিক উপায়**: `zi = zipfile.ZipInfo(path, date); zi.external_attr = (0o644 << 16); zi.compress_type = zipfile.ZIP_DEFLATED; z.writestr(zi, content)`। যাচাই: `[i.filename for i in infolist() if ((i.external_attr>>16)&0o777)==0]` → খালি হতে হবে। (`z.write(file, arcname)` এ এই সমস্যা নেই — ডিস্কের পারমিশন কপি হয়।)
> **নিয়ম**: `uploads/` এর আসল ছবি জিপে দেবেন না (বড়), কিন্তু **সাবফোল্ডারগুলো অবশ্যই রাখুন** (প্লেসহোল্ডার `index.html` দিয়ে, উপরের সঠিক পারমিশন সহ) — নাহলে নতুন ইনস্টলে ছবি আপলোড ভাঙে। (⚠️ PowerShell-এ literal backslash `'\\'` স্ক্রিপ্টে sandbox স্ক্যানার "Remove-Item on protected path" বলে ব্লক করে — Python পথই সহজ।)

## ⚠️ সবচেয়ে গুরুত্বপূর্ণ নিয়ম: টেস্টিং

1. **কখনো সরাসরি `educenter` (real) DB তে schema পরিবর্তন টেস্ট করা যাবে না।** সবসময় একটা আলাদা `educenter_test` DB বানিয়ে সেখানে schema.sql import করে টেস্ট করতে হবে।
2. যেকোনো schema পরিবর্তন (নতুন কলাম/টেবিল) `database/schema.sql` এ আপডেট করার পর:
   - `educenter_test` এ ফুল টেস্ট করুন (isolated copy `D:\xampp\htdocs\website_test` বানিয়ে)
   - সব ঠিক থাকলে `mysqldump` দিয়ে real `educenter` DB এর ব্যাকআপ নিন
   - তারপর real DB তে **non-destructive `ALTER TABLE`** (ড্রপ/রিক্রিয়েট না করে) দিয়ে পরিবর্তন apply করুন
   - ডেটা কাউন্ট (courses/registrations/admin_users) আগে-পরে মিলিয়ে নিশ্চিত হন কিছু হারায়নি
3. টেস্ট শেষে অবশ্যই `website_test` ফোল্ডার ও `educenter_test` DB ডিলিট করে ফেলুন (cleanup)।
4. প্রতিটা PHP ফাইল লেখার পর `/d/xampp/php/php.exe -l <file>` দিয়ে lint করুন।
5. Functional test curl দিয়ে — লগইনের জন্য cookie jar (`-c`/`-b`) লাগবে (CSRF সেশনের সাথে বাঁধা)।
6. **বাংলা টেক্সট curl দিয়ে POST করতে হলে `--data-urlencode "field@filepath"` (ফাইল থেকে পড়ে) ব্যবহার করুন, কমান্ড-লাইন আর্গুমেন্ট হিসেবে সরাসরি বাংলা টাইপ করবেন না** — Windows/Git Bash এ মাঝেমধ্যে মাল্টিবাইট UTF-8 আর্গুমেন্ট mangled হয়ে যায় (`?` তে পরিণত হয়)। ফাইলপাথ অবশ্যই Windows-style (`C:\Users\...`) দিতে হবে, Git Bash `/c/...` পাথ দিলে curl এরর দেয়।
7. **mysql CLI তে বাংলা টেক্সট আউটপুট `?????` দেখালে সেটা ডেটা করাপশন না হতে পারে** — Git Bash/mysql.exe টার্মিনাল Windows console codepage এর কারণে বাংলা ইউনিকোড রেন্ডার করতে পারে না। আসল ডেটা ঠিক আছে কিনা যাচাই করতে `SELECT HEX(column)` ব্যবহার করুন অথবা ব্রাউজারে রেন্ডারড HTML আউটপুট (curl দিয়ে পেজ ফেচ করে grep) চেক করুন।

### ⚠️ সাবধানতা: বাশ cwd রিসেট
মাঝে মাঝে বাশ টুলের cwd রিসেট হয়ে `D:\clude_project\website` তে ফিরে যায় (এটা টুল হারনেসের কারণে)। এর ফলে `curl -o file.html` জাতীয় কমান্ড ভুলবশত **আসল প্রজেক্ট ফোল্ডারে** টেস্ট ফাইল লিখে ফেলতে পারে — কাজ শেষে সবসময় `ls` দিয়ে stray `.html`/`.txt` ফাইল আছে কিনা চেক করুন এবং **নির্দিষ্ট নামে** ডিলিট করুন (`rm *.html` এর মতো broad glob কখনো ব্যবহার করবেন না — একবার এভাবে আসল `indexv3.html` ভুলে ডিলিট হয়ে গিয়েছিল, ব্যাকআপ থেকে recover করতে হয়েছিল)।

## আর্কিটেকচার

- **এন্ট্রি পয়েন্ট**: প্রতিটা top-level `.php` ফাইল নিজেই একটা পেজ (কোনো router নেই)
- **শেয়ার্ড কোড**: `includes/functions.php` (db, session, e(), csrf, slug, price parser ইত্যাদি), `includes/db.php` (PDO), `includes/upload.php`, `includes/bd-districts.php`, `includes/courier/*`
- **অ্যাডমিন CRUD**: `admin/includes/entities.php` এ একটা কনফিগ অ্যারে (courses/worksheets/products/teachers/reviews/notices/gallery/faqs) — `admin/manage.php` এই কনফিগ পড়ে জেনেরিকভাবে list/add/edit/delete হ্যান্ডেল করে। **নতুন কনটেন্ট টাইপ যোগ করতে শুধু entities.php এ এন্ট্রি যোগ করলেই হয়**, নতুন PHP ফাইল লাগে না। ফিল্ড কনফিগে এক্সট্রা ফ্ল্যাগ সাপোর্ট করে: `'suggest' => true` (text ফিল্ডে আগে ব্যবহৃত distinct মানের datalist দেখায়, ড্রপডাউনের মতো বেছে নেওয়া যায় কিন্তু ফ্রি-টেক্সটও থাকে), `'auto_next' => true` (number ফিল্ডে নতুন এন্ট্রিতে `MAX(col)+1` অটো বসে, এডিটে প্রযোজ্য না), এবং `'warn_off' => true` + `'toggle_label' => '...'` (checkbox ফিল্ড — যেমন `is_active` — এখন On/Off স্লাইড সুইচ হিসেবে রেন্ডার হয়; `warn_off` মার্ক করা থাকলে চালু→বন্ধ করার সময় `handleToggleWarn()` (`admin/includes/layout-bottom.php`) কনফার্মেশন মডাল দেখায়, চালু করার সময় না)। ফর্ম-রেন্ডারিং হেল্পার (`render_field()`, `admin_image_src()`, `make_unique_value()`) **`admin/includes/form-helpers.php`** এ শেয়ার্ড রাখা হয় (আগে `manage.php` এর ভেতরে ছিল) — `manage.php` ও কাস্টম পেজ (যেমন `course-batches.php`) দুটোই এই ফাইল require করে রিইউজ করে।
- **কোর্স parent/child (courses ↔ course_batches, ২০২৬-০৭-১২ থেকে)**: `courses` (`entities.php` তে জেনেরিক ইঞ্জিনে ম্যানেজড, শুধু `title`+`sort_order`) একটা হালকা "কোর্স আইডেন্টিটি" parent টেবিল। আসল বিক্রয়যোগ্য/দৃশ্যমান কনটেন্ট (slug/image/price/duration/instructor/batch_name/description/features/hide_parcel/registration_open/is_active — **সবগুলো ব্যাচ-ভিত্তিক**, কোর্স-ভিত্তিক না, এটা ইউজারের স্পষ্ট সিদ্ধান্ত) `course_batches` (child, FK→courses ON DELETE CASCADE, `UNIQUE(course_id, batch_name)`) এ থাকে — একটা কোর্সের একাধিক ব্যাচ থাকতে পারে, প্রতিটার সম্পূর্ণ আলাদা তথ্য। ডেডিকেটেড কাস্টম পেজ **`admin/course-batches.php`** (`?course_id=X`) এই child টেবিল CRUD করে (জেনেরিক `manage.php` ইঞ্জিন নেস্টেড parent/child সাপোর্ট করে না বলে) — batch_name ইনপুটে গ্লোবাল datalist সাজেশন কিন্তু ইউনিকনেস চেক শুধু সেই course_id এর মধ্যে, sort_order/auto_next-ও course_id-স্কোপড (গ্লোবাল না)। **পাবলিক সাইটে/রেজিস্ট্রেশনে `course_id` GET/POST প্যারামিটার এখন বাস্তবে `course_batches.id` বোঝায়** (এক্সটার্নাল নাম অপরিবর্তিত রাখা হয়েছে, ভেতরের কোয়েরি বদলেছে)। `course_features.batch_id` (আগে `course_id` নামে ছিল) এখন `course_batches` কে পয়েন্ট করে। `registrations.item_id` (type='course') সরাসরি `course_batches.id` পয়েন্ট করে — কিন্তু `admin/course-data.php`-এর কোর্স-পিকার dropdown ইচ্ছাকৃতভাবে courses/course_batches জয়েন করে না, বরং `registrations.item_id`/`item_title` (রেজিস্ট্রেশনের সময় নেওয়া স্ন্যাপশট) থেকে সরাসরি distinct list বানায় — তাই কোর্স/ব্যাচ পরে রিনেম/ডিলিট হলেও পুরনো রেজিস্ট্রেশনের ফিল্টার-অপশন অক্ষত থাকে। **নিয়ম**: title/price/is_active ইত্যাদি নিয়ে কোনো নতুন কোর্স-সম্পর্কিত কোয়েরি লিখলে মনে রাখবেন `courses` এ শুধু title, বাকি সব `course_batches` এ — `courses.php`/`index.php`/`course-register.php`/`course-register-submit.php`/`includes/functions.php`-এর `fetch_item()`/`get_item_details()` এবং `admin/courier.php`-এর `EXISTS (SELECT ... FROM course_batches ...)` সাবকোয়েরি — এই সবগুলোই `course_batches JOIN courses` (title আনতে) প্যাটার্ন অনুসরণ করে, নতুন কোয়েরিতেও একই প্যাটার্ন কপি করুন। মাইগ্রেশন স্ক্রিপ্ট (`database/migrate-course-batch-split.sql`, `RENAME TABLE courses TO course_batches` + নতুন slim parent তৈরি) real DB তে প্রয়োগ হয়ে গেছে — নতুন ইনস্টলে `schema.sql` সরাসরি নতুন স্ট্রাকচার দিয়ে তৈরি হয়, migrate স্ক্রিপ্ট শুধু ঐতিহাসিক রেফারেন্সের জন্য রাখা।
- **রেজিস্ট্রেশন/অর্ডার লিস্ট ফিল্টার+পেজিনেশন প্যাটার্ন**: `admin/registrations.php` এ একটা কেন্দ্রীয় `$activeFilters` অ্যারে (status/type/q/item/date_from/date_to, খালি মান বাদ) + `reg_url(array $overrides = [])` হেল্পার ফাংশন — এটা `$activeFilters`-এর সাথে override মার্জ করে সব ফিল্টার-লিংক/পেজিনেশন-লিংক/`return_url` তৈরি করে। নতুন ফিল্টার যোগ করলে শুধু `$activeFilters` অ্যারেতে key যোগ করলেই সব লিংকে অটো propagate হয় (আগে প্রতিটা লিংকে আলাদা `http_build_query(array_filter([...]))` কল ছিল, নতুন ফিল্টার যোগ করতে প্রতিটা জায়গা এডিট করা লাগত)। পেজিনেশন ২৫/পেজ (`LIMIT`/`OFFSET`), ফিল্টার বদলালে page override না দেওয়ায় স্বয়ংক্রিয়ভাবে পেজ ১-এ রিসেট হয়। **ভবিষ্যতে অন্য কোনো লিস্ট পেজে (course-data.php, download-logs.php, visitor-logs.php — এগুলো এখন ফিক্সড LIMIT/no-pagination) একই সুবিধা লাগলে এই প্যাটার্নটাই কপি করুন।** ফিল্টার ফর্মে বাটনে ক্লিক করা লাগে না — সার্চ বক্সে টাইপ করলে ৫০০ms debounce এর পর, আর item/date_from/date_to তে `onchange`-এ সরাসরি `document.getElementById('regFilterForm').submit()` কল হয় (ফুল GET রিলোড, কোনো AJAX না — এই কোডবেসে কোথাও AJAX-ভিত্তিক পার্শিয়াল রিফ্রেশ নেই, তাই এটাই কনভেনশনের সাথে সবচেয়ে মানানসই সমাধান)।
- **রেজিস্ট্রেশন/অর্ডার লিস্টে টাইপ-স্পেসিফিক কলাম**: `type` ফিল্টার অনুযায়ী তিনটা আলাদা টেবিল-লেআউট রেন্ডার হয় — `course` (course-register.php ফর্মের ঠিক যে ক্রমে ফিল্ড নেওয়া হয় সেই ক্রমে: আইটেম/ব্যাচ/শিশুর নাম/মোবাইল-মা/জন্ম তারিখ/ফেসবুক আইডি/মোবাইল-বাবা/রিসিভার নাম/রিসিভার নম্বর/ঠিকানা), `worksheet`/`product` (দুটোই register.php ফর্ম শেয়ার করে বলে একই লেআউট: মোবাইল/নাম/ইমেইল/ঠিকানা/আইটেম/পরিমাণ, batch নেই কারণ শুধু কোর্সেই প্রযোজ্য), আর "সব টাইপ" (ফিল্টার না করা থাকলে) এ একটা জেনেরিক মিশ্র-টাইপ লেআউট (নাম/ফোন/টাইপ/আইটেম/ব্যাচ/ঠিকানা)। প্রতিটাতেই শেষে স্ট্যাটাস/আয়/তারিখ/অ্যাকশন কমন। স্ট্যাটাস-সেল ও আয়-সেল রেন্ডারিং `reg_status_cell()`/`reg_income_cell()` হেল্পার ফাংশনে বের করা (তিনটা লেআউটেই ডুপ্লিকেট না করে রিইউজ)। **নিয়ম**: কোনো ফর্মে (course-register.php/register.php) নতুন ফিল্ড যোগ করলে registrations.php-এর সংশ্লিষ্ট কলাম-লেআউটেও যোগ করুন যাতে অ্যাডমিন লিস্টে "যা চাওয়া হয়েছে তাই দেখা যায়" নীতি বজায় থাকে।
    - **ব্যাচ ফিল্টার (কাস্কেডিং)**: `filterBatch` (registrations.batch, শুধু কোর্স) ফিল্টার আইটেম/টাইপের সাথে কাস্কেড করে — ব্যাচ ড্রপডাউনের অপশনগুলো `$filterItem` সিলেক্ট করা থাকলে সেই কোর্সের ব্যাচেই সীমাবদ্ধ থাকে (`WHERE type='course' AND item_title = :item`), কোনো আইটেম সিলেক্ট না থাকলে সব কোর্সের সব ব্যাচ দেখায়। আইটেম ড্রপডাউনের `onchange` এ ব্যাচ সিলেক্টের ভ্যালু JS দিয়ে খালি করে দেওয়া হয় সাবমিটের আগে (`document.getElementById('regBatchSelect').value=''`) — নাহলে আইটেম বদলালে আগের (এখন অপ্রাসঙ্গিক) ব্যাচ ভ্যালু সাবমিট হয়ে ভুলভাবে ০ রেজাল্ট দেখাত। টাইপ পিল-লিংকেও (`reg_url(['type'=>..., 'item'=>null, 'batch'=>null])`) একই কারণে batch ক্লিয়ার করা হয়।
- **কোর্স ডেটা টেবিল**: `admin/course-data.php` — জেনেরিক CRUD এর বাইরে একটা কাস্টম পেজ, শুধু কোর্স রেজিস্ট্রেশন (registrations টেবিলের type='course' রো) কোর্স+ব্যাচ অনুযায়ী গ্রুপড ভিউ ও ডুপ্লিকেট ডিটেকশনের জন্য। এডিট/ডিলিট/ভিউ অ্যাকশন `admin/registrations.php` এর শেয়ার্ড handler গুলোই রিইউজ করে (নতুন CRUD লজিক ডুপ্লিকেট করেনি)।

WHERE registration_id = r.id AND ...)` সাবকোয়েরি দিয়ে হয়, অর্থাৎ registration এর **যেকোনো একটা** ব্যাচ ম্যাচ করলেই সেই রেজিস্ট্রেশন দেখায়) `courier_url(array $overrides = [])` হেল্পার দিয়ে (registrations.php এর `reg_url()` প্যাটার্নের কপি)। **কাস্কেডিং**: "নির্দিষ্ট আইটেম" বাছাই করলে কোর্স-ব্যাচ ও ব্যাচ-লেবেল ড্রপডাউনের অপশন সেই আইটেমেই সীমাবদ্ধ হয়ে যায় (`$filterPeriodLabelOptions` — item-scoped, filter dropdown-এর জন্য; এটা `$periodLabelSuggestions` থেকে আলাদা রাখা হয়েছে, যেটা গ্লোবাল এবং "নতুন ব্যাচ" ফর্মের datalist সাজেশনে ব্যবহার হয়), আইটেম বদলালে JS দিয়ে course_batch/period_label সিলেকশন অটো-ক্লিয়ার হয় (`courierItemSelect` এর onchange)। checkbox + দুইটা বাল্ক অ্যাকশন — "⚡ দ্রুত পাঠান" (কোনো কাস্টম লেবেল ছাড়া, ডিফল্ট দিয়ে সরাসরি পাঠায়) ও "🗓️ নতুন ব্যাচ" (মডাল খুলে period_label/item_description/amount_to_collect ইনপুট নেয়, "খসড়া তৈরি করুন" বা "তৈরি করে পাঠান" — ফিল্টার করে একটা কোর্স-ব্যাচ/আইটেম বেছে নিয়ে সবার জন্য একসাথে "পরের মাস" ব্যাচ তৈরির মূল ব্যবহার-ক্ষেত্র)। `admin/bulk-courier-action.php` (আগে `bulk-send-to-courier.php` নামে ছিল, রিনেম করা হয়েছে) `mode=draft|send` + শেয়ার্ড ওভাররাইড ফিল্ড নিয়ে প্রতিটা সিলেক্ট করা registration এ লুপ করে `save_courier_batch()`/`send_courier_batch()` কল করে। ডিটেইল ভিউ (`?action=view&id=X`) এ ব্যাচ ইতিহাস টেবিল + প্রতিটা ব্যাচের এডিট/ডিলিট লিংক + "নতুন ব্যাচ" বাটন (`?batch=new` নতুন, `?batch=<id>` এডিট, প্যারামিটার ছাড়া সর্বশেষ ব্যাচ এডিট মোডে খোলে ডিফল্ট হিসেবে)। **`courier_order_data` টেবিল এখন legacy/অব্যবহৃত** (courier_batches দিয়ে প্রতিস্থাপিত, ড্রপ করা হয়নি শুধু পুরনো ডেটা সংরক্ষণের জন্য — কোড থেকে আর পড়া/লেখা হয় না)।

## ⚠️⚠️ আয়ের হিসাব (income) — টাকা জড়িত, খুব সাবধানে ছোঁবেন

**কেন্দ্রীয় ফাংশন**: `sync_income_for_status(PDO $db, array $reg, string $newStatus)` (`admin/registrations.php`)। status `confirmed`/`shipped`/`delivered` (`INCOME_STATUSES`) হলে `income` টেবিলে রো + `registrations.income_approved=1`; এর বাইরে গেলে আয়ের রো মুছে `income_approved=0`।

- **🔴 পরিমাণ বের করার ক্রম (২০২৬-০৭-২০ এ অডিটে বাগ ধরা পড়ার পর)**: (১) আইটেমের **বর্তমান দাম** (`course_batches`/`worksheets`/`products` — `item_id` থেকে) × quantity; (২) না পেলে **`registrations.income_amount` স্ন্যাপশট** (আগের অনুমোদনের পরিমাণ); (৩) তাও না পেলে **flash error দিয়ে অ্যাডমিনকে জানানো** — নীরবে ছেড়ে দেওয়া যাবে না। **আগের বাগ**: আইটেম ডিলিট/আর্কাইভ হয়ে গেলে দাম পাওয়া যেত না → `$amount = 0` → `if ($amount > 0)` স্কিপ → রেজিস্ট্রেশন "কনফার্ম" দেখাত কিন্তু **আয় নীরবে হারিয়ে যেত**।
- **🔴 `income_amount` কখনো NULL করবেন না** আয় বাদ দেওয়ার সময় — শুধু `income_approved = 0, approved_at = NULL` করুন। `income_amount` "সর্বশেষ জানা পরিমাণ" হিসেবে থেকে যায়, যেটা উপরের ধাপ (২)-এর ভিত্তি। প্রদর্শনে সমস্যা নেই কারণ তালিকা (`reg_income_cell()`) ও ডিটেইল দুটোই `income_approved` দেখে গেট করে। (দুই জায়গায় এই আপডেট আছে: `sync_income_for_status()` ও `action=unapprove-income` হ্যান্ডলার — একটা বদলালে অন্যটাও দেখুন।)
- **`income.registration_id` FK = `ON DELETE SET NULL`** (cascade **না**, ইচ্ছাকৃত)। তাই যেকোনো জায়গা থেকে রেজিস্ট্রেশন ডিলিট করলে **আগে `DELETE FROM income WHERE registration_id = ...` চালাতে হবে, তারপর রেজিস্ট্রেশন** — নাহলে আয়ের রো `registration_id = NULL` নিয়ে অনাথ হয়ে বইয়ে থেকে যাবে (মোট আয় ভুল দেখাবে, উৎসও খুঁজে পাওয়া যাবে না)। **তিনটা ডিলিট-পথেই এই ক্রম মানা আছে**: `registrations.php?action=delete`, `courier.php?action=delete`, আর `course-data.php` (registrations.php-এর হ্যান্ডলার রিইউজ করে)। **নতুন কোথাও রেজিস্ট্রেশন ডিলিট যোগ করলে এই ক্রম + তার আগে `archive_entity()` — তিনটাই লাগবে।**
- **আর্কাইভ/রিস্টোর**: `income` হলো registrations-এর archive child (`archive_children_map()`), তাই ডিলিটে আয় সহ বান্ডল হয় আর রিস্টোরে আসল id ও পরিমাণ সহ ফেরে (E2E যাচাইকৃত — ডুপ্লিকেট হয় না)।
- **টেস্টের নিয়ম**: আয়-সংক্রান্ত যেকোনো পরিবর্তন **isolated `educenter_test` + `website_test` কপিতে** যাচাই করুন (real DB-র `mysqldump` ইমপোর্ট করে), তারপর প্রতিটা ধাপে DB-র `SUM(amount)` ও অনাথ-কাউন্ট মিলিয়ে দেখুন। যাচাই করার তালিকা: confirm→cancel→confirm (দুইবার), ডিলিট→রিস্টোর, এবং income.php/expenses.php/ড্যাশবোর্ড তিন জায়গার সংখ্যা DB-র সাথে মেলা।

## ⚠️ ছবি আপলোড (`includes/upload.php`) — লাইভে যেসব ফাঁদে পড়েছিলাম

- **`is_writable()` বিশ্বাস করবেন না** — কিছু শেয়ার্ড হোস্টে `true` বলে অথচ আসল লেখা ব্যর্থ হয়। `upload_dir_is_writable()` ব্যবহার করুন (আসল ফাইল লিখে-মুছে যাচাই)। ব্যর্থ হলে কোড নিজেই `@chmod(0755)` দিয়ে সারানোর চেষ্টা করে, তারপরও না হলে **File Manager-এ ঠিক কী করতে হবে** সেটা বলে দেয়।
- **`mkdir()` এর রিটার্ন ভ্যালু চেক করুন** (আগে করা হতো না — ফোল্ডার তৈরি না হলেও কোড এগিয়ে গিয়ে অস্পষ্ট এররে ঠেকত), আর তৈরির পরে `@chmod(0755)` (umask-সুরক্ষা)।
- **GD মেমরি গার্ড**: মোবাইলের ৪০০০×৩০০০ ছবি GD-তে খুলতে ~৪৮ MB লাগে; `memory_limit` কম হলে `imagecreatefrom*()` **fatal error** দিয়ে পুরো রিকোয়েস্ট মেরে ফেলে (catch করা যায় না, সাদা পেজ)। `resize_image_to_canvas()` তাই আগে হিসাব করে — **সোর্স + ফিক্সড ১০০০×৭৫০ ক্যানভাস + ওভারহেড** (ক্যানভাসটা বাদ দিলে ছোট ছবিতেও ক্র্যাশ হয়, একবার হয়েছিল) — কুলাবে না মনে হলে `false` দিয়ে `move_uploaded_file()` fallback-এ যায় (রিসাইজ হয় না, কিন্তু আপলোড সফল)।
- **ব্যর্থতার কারণ নির্ণয়**: `upload_failure_reason()` তিনটা আসল পরীক্ষা চালায় — tmp ফাইল আছে? ১ বাইট লেখা যায় (পারমিশন)? **ছবির সমান সাইজ** লেখা যায় (ডিস্ক কোটা)? ⚠️ `disk_free_space()` **অকেজো** — সেটা পুরো সার্ভারের খালি জায়গা দেখায়, cPanel অ্যাকাউন্টের কোটা না।

- **🔴 হোস্ট-ভিত্তিক যাচাই**: `is_facebook_url()` অবশ্যই `parse_url(..., PHP_URL_HOST)` দিয়ে চেক করবে — `strpos($url, 'facebook.com')` দিয়ে করলে `https://evil.com/?facebook.com` পাস করে যেত (ইউনিট-টেস্টে ধরা)।

- **নতুন পাবলিক সেকশন বানালে**: `render_page_header()` এর সিগনেচার **`(icon, eyebrow, title, subtitle, accentClass = 'text-gray-800')` — প্রথম ৪টা আবশ্যক**। কম দিলে `ArgumentCountError` হয়ে **পেজ ঐ জায়গা থেকে ভেঙে যায়** (ফুটার সহ বাকিটা আসে না, HTTP তবু 200) — একবার হয়েছিল, Apache error log থেকে ধরা পড়ে। নতুন পেজ/সেকশনের পর `</html>` রেন্ডার হচ্ছে কিনা curl দিয়ে দেখুন।
- **🔴 SQL মাইগ্রেশনে বাংলা টেক্সট লিখবেন না** — কিছু টুলে (বিশেষত Windows-এর `mysql.exe` CLI) ডাবল-এনকোড হয়ে mojibake হয়ে যায় (`C393C2AA...`)। বাংলা ডিফল্ট লেখা **PHP-তে** `get_setting('key') ?: 'বাংলা ডিফল্ট'` ফলব্যাক হিসেবে রাখুন, SQL-এ খালি স্ট্রিং বসান। যাচাই: `SELECT HEX(col)` — সঠিক বাংলা `E0A6...`/`E0A7...` দিয়ে শুরু হয়। (লোকালে বাংলা ডেটা বসাতে হলে mysql CLI নয়, PDO/PHP ব্যবহার করুন — `charset=utf8mb4` সহ।)
- **🔴🔴 ফেসবুকের এমবেডেড পোস্টের ন্যূনতম প্রস্থ ৩৫০px** — এর কম `width=` চাইলে প্লাগইন `400 Bad Request` + `X-Frame-Options: DENY` ফেরত দেয়, ব্রাউজার দেখায় **"www.facebook.com refused to connect"**। (২০২৬-০৭-২০ এ লাইভে: PC-তে কার্ড চওড়া বলে চলত, মোবাইলে ~৩১৯px হওয়ায় ভাঙত। curl দিয়ে desktop/mobile UA পরীক্ষা করে নিশ্চিত হওয়া গেছে ফেসবুক **মোবাইল UA ব্লক করে না** — সমস্যাটা নিছক প্রস্থের।) **সমাধান**: সবসময় ≥৩৫০ চাওয়া হয়, কন্টেইনার সরু হলে CSS `transform: scale()` দিয়ে ছোট করে বসানো হয় (`transform-origin: top left`), আর মোড়কের উচ্চতা JS নিজে `FB_H * scale` বসায় — নাহলে নিচে বড় ফাঁকা জায়গা থাকত।
- **অতিরিক্ত সাদা জায়গা**: ফেসবুকের এমবেড সর্বোচ্চ ৫০০px। কলাম তার চেয়ে চওড়া হলে দুই পাশে ফাঁকা পড়ে — তাই কার্ডে `max-w-[540px]` + গ্রিডে `justify-items-center`।
- **🔴 এমবেডের প্রস্থ**: ফেসবুকের এমবেড **fluid নয়** — URL-এর `width=` এ যা দেওয়া হয় ঠিক সেই মাপে রেন্ডার করে, CSS দিয়ে ছোট-বড় করা যায় না (cross-origin iframe)। হার্ডকোড প্রস্থ কন্টেইনারের চেয়ে বড় হলে **লেখা কেটে যায়**। তাই `site-footer.php`-এর lazy-loader কন্টেইনার মেপে (২৮০–৫০০ ক্ল্যাম্প) সেই মানটাই URL-এ বসায় ও iframe-এ ঠিক ঐ px প্রস্থ দেয়; রিসাইজে ৪০px+ বদলালে debounced রি-লোড। পোস্ট গ্রিড `xl:grid-cols-2` (তার নিচে ১ কলাম) — সরু কার্ডে ফেসবুকের পোস্ট পড়া যায় না।
- **iframe-এর ভেতরের কিছু নিয়ন্ত্রণ করা যায় না** ("See more", লাইক বাটন ইত্যাদি — cross-origin)। ইউজারকে পুরো পোস্টে পাঠাতে **নিজের** একটা লিংক দিন (`.fb-open-link`) — মোবাইলে সাধারণ `https://facebook.com/...` লিংক OS নিজেই ফেসবুক অ্যাপে খোলে (App Links / Universal Links)। **`fb://` ডিপ-লিংক ব্যবহার করবেন না** — অ্যাপ ইনস্টল না থাকলে ভেঙে যায়।
- **কোন লিংক কাজ করে**: পোস্টের **আসল পারমালিংক** (তারিখে ক্লিক করে অ্যাড্রেস বার থেকে নেওয়া)। **`/share/p/...` শর্ট-লিংক কাজ করে না** — "This Facebook post is no longer available" দেখায়। **রিল ফেসবুক আনুষ্ঠানিকভাবে embed সাপোর্ট করে না।** পোস্ট অবশ্যই Public।
- **`'help' => '...'`** এখন যেকোনো এন্টিটি-ফিল্ডে দেওয়া যায় (`admin/includes/form-helpers.php`) — ইনপুটের নিচে ছোট ধূসর ব্যাখ্যা বসে। `render_field()` এখন পাতলা র‍্যাপার, আসল ইনপুট-মার্কআপ `render_field_input()`-এ। ইউজারের কোডিং জ্ঞান নেই বলে নতুন ফিল্ডে এই ইঙ্গিত দেওয়া ভালো।



> **🔴 কোড/CSS ব্লক range দিয়ে মুছলে সাবধান (২০২৬-০৭-২১ এ শেখা)**: `substr`/`sed` দিয়ে দুই anchor-এর মাঝের অংশ মুছলে **মাঝখানে অন্য ব্লক থাকলে সেটাও মুছে যায়** (একবার `#back-to-top` সরাতে গিয়ে মাঝের `.fb-card` CSS ৬৬ লাইন হারিয়েছিল)। মোছার আগে ঐ range-এ আর কী আছে দেখুন; git থাকায় `git show <commit>~1:path` দিয়ে দ্রুত ফেরানো যায়।

## ⚠️ পুনরাবৃত্ত বাগ প্যাটার্ন: variable collision

PHP `require`/`include` একই global scope শেয়ার করে। শেয়ার্ড partial ফাইলের ভেতরে `foreach` লুপে সাধারণ ভ্যারিয়েবল নাম (`$item`, `$conf`, `$key`, `$row`) ব্যবহার করলে সেটা calling script এর একই নামের ভ্যারিয়েবল **নিঃশব্দে ওভাররাইট** করে দেয় (কোনো error ছাড়াই)। এই বাগ ৩ বার ধরা পড়েছে ও ফিক্স হয়েছে:
- `layout-top.php` এর সাইডবার লুপে `$conf` → `admin/manage.php` এর `$conf` ক্লোবার করছিল
- `site-header.php` এর nav লুপে `$item` (২ বার, desktop+mobile) → `detail.php`/`register.php` এর `$item` ক্লোবার করছিল

**নিয়ম**: শেয়ার্ড partial ফাইলে লুপ ভ্যারিয়েবল সবসময় distinctive/prefixed নাম দিন (যেমন `$navItem`, `$navEntityConf`), সাধারণ নাম না।

## ⚠️ PDO non-emulated prepare: এক্সট্রা bound parameter দিলে fatal crash

`includes/db.php` এ `PDO::ATTR_EMULATE_PREPARES => false` সেট করা আছে (নিরাপত্তার জন্য, native prepared statements)। এর একটা সাইড-ইফেক্ট: `execute($params)` এ যদি এমন কোনো key থাকে যেটা SQL কুয়েরিতে `:key` হিসেবে ব্যবহৃতই হয়নি, তাহলে **নিঃশব্দে ignore হয় না — সরাসরি `PDOException: SQLSTATE[HY093]: Invalid parameter number` ছুঁড়ে পুরো পেজ ভেঙে দেয়**। এই বাগ ধরা পড়েছিল `CourierManager.php`-এর `save_courier_batch()` এ — একটা `$params` অ্যারে বানিয়ে সেটা INSERT ও UPDATE দুটো ভিন্ন কুয়েরিতে reuse করা হচ্ছিল (INSERT এ `:reg_id` লাগে, UPDATE এ `:id` লাগে, কিন্তু দুটোতেই একই অ্যারে পাঠানো হচ্ছিল যাতে অন্যটার জন্য একটা এক্সট্রা key থেকে যাচ্ছিল) — isolated টেস্টে (curl দিয়ে সেভ+পাঠানোর ফ্লো টেস্ট করার সময়) ধরা পড়ে, রিয়েল DB তে migration/apply করার আগেই ফিক্স হয়েছে। **নিয়ম**: একই `$params` অ্যারে একাধিক ভিন্ন প্রিপেয়ার্ড স্টেটমেন্টে সরাসরি reuse করবেন না যদি তাদের placeholder সেট ভিন্ন হয় — হয় প্রতিটার জন্য আলাদা অ্যারে বানান, নাহলে common ফিল্ডগুলো একটা বেস অ্যারেতে রেখে প্রতিটা কুয়েরির আগে query-specific key `+`/`array_merge` করে যোগ করুন (`$fieldParams + ['id' => ...]` প্যাটার্ন)।

## ⚠️ গুরুত্বপূর্ণ সীমাবদ্ধতা: কোনো লাইভ ব্রাউজার প্রিভিউ নেই

এই প্রজেক্টে এখন পর্যন্ত (এই লেখা পর্যন্ত) কোনো সেশনেই লাইভ ব্রাউজার প্রিভিউ/স্ক্রিনশট টুল (Claude Preview, chrome extension, computer-use) কানেক্ট করা যায়নি। তাই **সব টেস্টিং curl + grep দিয়ে HTML সোর্স চেক করে করা হয়েছে — কখনো আসল রেন্ডারিং visually verify করা হয়নি।** এর ফলে একবার একটা real bug (CSS caching এর কারণে পুরো হোমপেজ ভেঙে দেখাচ্ছিল) কয়েক রাউন্ড কাজ করার পরও ধরা পড়েনি, যতক্ষণ না ইউজার নিজে স্ক্রিনশট পাঠালেন।

**পরিণতি**: CSS/JS এর মতো স্ট্যাটিক অ্যাসেট পরিবর্তনের পর সবসময় cache-busting নিশ্চিত করুন (`assets/css/style.css` এখন `includes/site-header.php` তে `filemtime()` দিয়ে অটো-ভার্সনড, নতুন কোনো CSS/JS ফাইল যোগ করলে একই প্যাটার্ন অনুসরণ করুন)। ভিজ্যুয়াল/লেআউট পরিবর্তনের পর ইউজারকে **হার্ড রিফ্রেশ** (Ctrl+Shift+R) করে দেখতে বলুন, এবং সম্ভব হলে প্রতিটা সেশনের শুরুতে preview টুল কানেক্ট আছে কিনা চেক করুন (`mcp__Claude_Preview__preview_list`, `mcp__claude-in-chrome__list_connected_browsers`)।

**আরেকটা উদাহরণ**: `admin/manage.php` এর এডিট ফর্মে ছবির প্রিভিউ (`<img src="uploads/courses/xxx.jpg">`) Phase 3 থেকেই ভাঙা ছিল (404) — DB তে সাইট-রুট-রিলেটিভ path সেভ থাকে, কিন্তু `admin/` সাবফোল্ডার থেকে সরাসরি ব্যবহার করলে ব্রাউজার `admin/uploads/...` খুঁজতো। curl+grep দিয়ে `<img>` ট্যাগটা ঠিকই দেখা যেত, কিন্তু আসলে রেন্ডার হতো কিনা কখনো ভেরিফাই হয়নি। **নিয়ম**: `admin/*.php` থেকে DB-তে সেভ করা যেকোনো ছবির path ব্যবহার করার সময় `admin_image_src()` (`admin/includes/form-helpers.php`) হেল্পার দিয়ে রেন্ডার করুন (রিলেটিভ হলে `../` যোগ করে, এক্সটার্নাল URL হলে অপরিবর্তিত রাখে) — সরাসরি `$row['image']` আউটপুট করবেন না। প্রতিটা admin ফাইল থেকে DB-স্টোরড ছবি/আপলোড path ব্যবহার করার সময় path resolution টা curl দিয়ে না, HTTP status code চেক করে (`curl -o /dev/null -w '%{http_code}'`) ভেরিফাই করুন।

## ⚠️ হেডিং টেক্সটে গ্রেডিয়েন্ট (`bg-clip-text`) ব্যবহার করবেন না

## ⚠️ `get_setting($key, $default)` এর ডিফল্ট প্যারামিটার খালি স্ট্রিং কে কভার করে না

`get_setting()` `??` (null coalescing) ব্যবহার করে — DB তে `setting_value = ''` (খালি স্ট্রিং, key আছে কিন্তু খালি) সেভ করা থাকলে `$default` আর ব্যবহার হয় না, খালি স্ট্রিংই রিটার্ন হয়। এই কারণে একবার About পেজ ফিচারে সেটিংস ফর্ম একবার (আংশিক খালি অবস্থায়) সাবমিট হওয়ায় লাইভ সাইটে "আমাদের মিশন"/"আমাদের দল" সেকশন ফাঁকা দেখাচ্ছিল, যদিও কোডে ডিফল্ট টেক্সট দেওয়া ছিল। **নিয়ম**: "সেভ করা না থাকলে/খালি থাকলে ডিফল্ট দেখাও" — এই প্যাটার্ন দরকার হলে `get_setting($key, $default)` এর তৃতীয় প্যারামিটারের উপর ভরসা না করে `get_setting($key) ?: $default` (empty-string-safe) ব্যবহার করুন, অথবা `about_page_defaults()`/`get_about_setting()` (`includes/functions.php`) এর মতো একটা ডেডিকেটেড হেল্পার বানান যেটা DRY-ও থাকে (about.php ও admin/settings.php দুটোতেই একই ডিফল্ট ব্যবহার হয়) এবং এই ফাঁদও এড়ায়।

## ⚠️ `make_slug()` uniqueness যাচাই করে না — auto_from ফিল্ড DB এ UNIQUE হলে সাবধান

`make_slug()` (`includes/functions.php`) শুধু টেক্সট থেকে slug ফরম্যাট বানায় (lowercase, hyphen), **কখনো DB চেক করে না** যে সেই slug আগে থেকে আছে কিনা। `course_batches.slug` কলামে DB-লেভেলে `UNIQUE` constraint আছে বলে, দুইটা ব্যাচের টাইটেল+batch_name থেকে একই slug জেনারেট হলে সরাসরি SAVE এ PDOException crash হয়। **নিয়ম**: `'auto_from' => '...'` মার্ক করা যেকোনো ফিল্ড যদি DB তে UNIQUE হয়, সেভ করার সময় অবশ্যই `make_unique_value($db, $table, $column, $baseValue, $excludeId)` (`admin/includes/form-helpers.php`) দিয়ে কনফ্লিক্ট চেক করে দরকার হলে `-2`/`-3` সাফিক্স যোগ করতে হবে — শুধু `make_slug()` কল করাই যথেষ্ট না। ভবিষ্যতে নতুন কোনো entity তে UNIQUE কলাম যোগ করলে এই প্যাটার্ন মনে রাখবেন।

## ডাটাবেস (সংক্ষেপে, বিস্তারিত schema.sql এ)

মূল টেবিল: `courses` (parent, শুধু title+sort_order), `course_batches` (child, আসল কোর্স-ব্যাচ কনটেন্ট — নিচে দেখুন), `course_features` (batch_id দিয়ে course_batches কে পয়েন্ট করে), `worksheets`, `products`, `product_features`, `teachers`, `reviews`, `notices`, `gallery`, `faqs`, `settings` (key-value), `admin_users`, `login_attempts`, `phone_lookup_attempts`, `form_submit_attempts` (পাবলিক ফর্ম রেট-লিমিট), `registrations`, `courier_shipments`, `payment_methods` (থ্যাংক-ইউ পেজের পেমেন্ট নাম্বার/WhatsApp, per-course scope সহ), `finance_categories`, `income`, `expenses`, `archived_items` (ডিলিট করা কনটেন্টের JSON বান্ডল — আর্কাইভ/রিস্টোর, উপরে দেখুন)

গুরুত্বপূর্ণ কলাম:
- `course_batches.hide_parcel` — Yes হলে কোর্স রেজিস্ট্রেশন ফর্মে রিসিভার তথ্য হাইড (ব্যাচ-ভিত্তিক, একই কোর্সের অন্য ব্যাচ প্রভাবিত হয় না)
- `course_batches.batch_name` — ব্যাচের নাম (যেমন "৫ম ব্যাচ"/"July_26"), NOT NULL, `UNIQUE(course_id, batch_name)` — `admin/course-batches.php` থেকে এডিট হয়
- `registrations.batch` — কোর্স রেজিস্ট্রেশনের সময় `course_batches.batch_name` থেকে স্ন্যাপশট নেওয়া হয় (`course-register-submit.php`) — ব্যাচের নাম পরে বদলালেও পুরনো রেজিস্ট্রেশনের ব্যাচ অপরিবর্তিত থাকে। ডুপ্লিকেট রেজিস্ট্রেশন ডিটেকশনে ব্যবহার হয় (`admin/course-data.php` — একই item_id+batch+customer_name+phone)
- `course_batches.registration_open` — `is_active` থেকে আলাদা: is_active ব্যাচ সাইটে দেখাবে/হাইড করবে, registration_open নিয়ন্ত্রণ করে ঐ নির্দিষ্ট ব্যাচের রেজিস্ট্রেশন ফর্ম খোলা/বন্ধ (বন্ধ হলে ব্যাচ দেখাবে কিন্তু "রেজিস্ট্রেশন বন্ধ" দেখাবে, ফর্ম আসবে না, সার্ভার-সাইডেও ব্লক করা আছে `course-register-submit.php` তে)
- `registrations.item_id` (type='course') — `course_batches.id` পয়েন্ট করে (courses.id না)
- `registrations.income_approved`/`income_amount`/`approved_at` — status confirmed/shipped/delivered হলে অটো সেট হয় (`sync_income_for_status()` ফাংশন, `admin/registrations.php` এ)
- `registrations` এ course-specific কলাম: `date_of_birth`, `facebook_id`, `father_mobile`, `receiver_name`, `receiver_phone` (worksheet/product এর জন্য NULL থাকে, generic `district`/`thana` ব্যবহার হয় না কোর্সে)

## নিরাপত্তা (ইতিমধ্যে ইমপ্লিমেন্টেড)
Brute-force protection (login_attempts, ৫/১৫মিন), CSRF সব ফর্মে, session hardening (HttpOnly/SameSite/Secure), **অ্যাডমিন নিষ্ক্রিয়তা-টাইমআউট (`ADMIN_IDLE_TIMEOUT_MINUTES=30`, `admin/includes/auth.php` — `admin_require_login()` এ sliding idle timeout, এক্সপায়ারে `login.php?expired=1`)**, security headers, `includes/` ফোল্ডার ব্লক (.htaccess), HTTPS enforcement (DEV_MODE=false এ), ফোন-লুকআপ রেট-লিমিট, ফাইল আপলোড MIME-check + random filename। **পাবলিক ফর্ম স্প্যাম-প্রোটেকশন** (২০২৬-০৭-১৫): রেজিস্ট্রেশন/অর্ডার ফর্মে honeypot (`website` হিডেন ফিল্ড) + টাইমিং (`form_ts`, ৩s এর কমে বট) + IP রেট-লিমিট (`form_submit_attempts` টেবিল, ৮/১০মিন) — হেল্পার `spam_protection_fields()`/`is_spam_submission()`/`form_submit_rate_limited()`/`form_record_submit()`/`client_ip()` (functions.php); ফর্মে `csrf_field()` এর পাশে `spam_protection_fields()`, submit স্ক্রিপ্টে CSRF-এর পর চেক।
